# contributing to dumpspot

dumpSpot is open to contributions that are thoughtful, minimal, and aligned with the product’s intent.

it is built to be simple, direct, and useful. every contribution should move it in that direction.

this document may evolve over time, but these are the general principles.

---

## before you start

it helps to quickly go through these:

- [`DESIGN.md`](./DESIGN.md) — visual system, spacing, typography, interaction patterns
- [`DEVELOPER.md`](./DEVELOPER.md) — structure, logic, and how things are put together
- this document — how contributions work

this will give you a good sense of how and why things are done a certain way.

---

## where to begin

- you can explore open issues here: [https://github.com/gauripod/dumpspot.info/issues](https://github.com/gauripod/dumpspot.info/issues)
- pick something small and clear to start
- if something feels unclear, opening an issue or discussion first is always welcome

---

## how the project is built

this project intentionally stays close to the basics of the web:

- html
- css
- javascript

no frameworks, no build tools.

it is built on this.

you are free to explore complex ideas and approaches, but the core remains simple and direct. contributions that respect this tend to fit best.

---

## what kind of contributions fit well

- fixing bugs
- improving performance
- making things more accessible
- refining ui while staying within the design system
- small, meaningful additions

---

## general approach

- try to keep changes focused and easy to follow
- follow the existing style and structure where possible
- test things locally before opening a pr
- be mindful of map, list, and form flows

if a change significantly alters structure or direction, it is best to discuss it first.

---

## pull requests

- a clear title and short explanation go a long way
- mention what you changed and why
- screenshots or recordings help a lot for ui changes
- keeping one idea per pr makes review easier

prs that follow these guidelines are more likely to be reviewed and accepted smoothly.

---

## review process

- pull requests are reviewed once a week
- reviews usually happen on **thursday evening**

feedback will be shared if something needs adjustment before merging.

---

## design alignment

the project has a defined visual language.

before making ui changes:

- refer to [`DESIGN.md`](./DESIGN.md)
- try to reuse existing patterns
- introduce new styles only when truly needed

---

## philosophy

a few ideas guide everything here:

- no friction
- no noise
